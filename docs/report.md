## 1. Вступ

### 1.1. Актуальність теми

У сучасному світі автономне обладнання, що працює від акумуляторних батарей, стає все більш розповсюдженим. Це включає безперебійні джерела живлення (UPS), резервні системи живлення, портативні пристрої, електротранспорт та багато іншого. Ефективний моніторинг стану батарей є критично важливим для забезпечення надійності таких систем.

### 1.2. Мета роботи

Розробити автоматизовану систему управління (АСУ) для моніторингу стану батарей автономного обладнання, що включає:
- Збір параметрів акумуляторів
- Зберігання історії вимірювань
- Моніторинг стану батарей в реальному часі
- Виявлення аварійних ситуацій
- Формування аналітичних звітів

### 1.3. Задачі роботи

1. Проаналізувати предметну область моніторингу батарей
2. Спроєктувати базу даних для зберігання інформації
3. Розробити backend на FastAPI
4. Розробити frontend на React
5. Реалізувати систему звітності
6. Провести тестування системи

### 1.4. Технологічний стек

- **Backend:** Python, FastAPI, SQLAlchemy, PostgreSQL
- **Frontend:** React, TypeScript, Tailwind CSS, Recharts
- **Testing:** Pytest, Vitest, React Testing Library
- **Migration:** Alembic

## 2. Аналіз предметної області

### 2.1. Предметна область

Система моніторингу батарей охоплює наступні аспекти:

**Параметри батарей, що відстежуються:**
- Рівень заряду (charge level) — відсоток від максимальної ємності
- Напруга (voltage) — електричний потенціал батареї
- Струм (current) — сила електричного струму
- Температура (temperature) — температура корпусу батареї

**Типи обладнання:**
- UPS (Uninterruptible Power Supply)
- Сонячні панелі з акумуляторами
- Електротранспорт
- Портативні генератори
- Резервні системи живлення

### 2.2. Користувачі системи

**Оператор**
- Додає обладнання та батареї
- Вводить вимірювання параметрів
- Переглядає поточні показники

**Інженер**
- Аналізує параметри батарей
- Переглядає історію вимірювань
- Формує звіти

**Адміністратор**
- Керує обладнанням та користувачами
- Редагує дані
- Керує базою даних

### 2.3. Вимоги до системи

**Функціональні вимоги:**
- CRUD операції для всіх сутностей
- Автоматичне виявлення критичних ситуацій
- Генерація аналітичних звітів
- Візуалізація даних у вигляді графіків

**Нефункціональні вимоги:**
- Швидкий час відгуку API (< 200ms)
- Масштабованість до 1000+ батарей
- Надійність зберігання даних
- Зручний інтерфейс користувача

## 3. Проєктування бази даних

### 3.1. Концептуальна модель

База даних складається з чотирьох основних сутностей:

```
┌─────────────┐       ┌─────────────┐       ┌─────────────┐       ┌─────────────┐
│  Equipment  │───────│  Batteries  │───────│Measurements │───────│  Incidents  │
│  (Обладнання)│  1:N  │ (Акумулятори)│  1:N  │(Вимірювання) │  1:N  │ (Інциденти) │
└─────────────┘       └─────────────┘       └─────────────┘       └─────────────┘
```

### 3.2. Логічна модель

**Таблиця Equipment (Обладнання)**
| Поле | Тип | Опис |
|------|-----|------|
| id | INTEGER PK | Унікальний ідентифікатор |
| name | VARCHAR(100) | Назва обладнання |
| type | VARCHAR(50) | Тип (UPS, Solar, EV тощо) |
| location | VARCHAR(100) | Розташування |
| status | VARCHAR(20) | Статус (active, inactive, maintenance) |
| created_at | TIMESTAMP | Дата створення |

**Таблиця Batteries (Акумулятори)**
| Поле | Тип | Опис |
|------|-----|------|
| id | INTEGER PK | Унікальний ідентифікатор |
| equipment_id | INTEGER FK | Посилання на обладнання |
| serial_number | VARCHAR(50) | Серійний номер |
| capacity | DECIMAL | Ємність (А·год) |
| voltage_nominal | DECIMAL | Номінальна напруга (В) |
| install_date | DATE | Дата встановлення |
| status | VARCHAR(20) | Статус батареї |

**Таблиця Measurements (Вимірювання)**
| Поле | Тип | Опис |
|------|-----|------|
| id | INTEGER PK | Унікальний ідентифікатор |
| battery_id | INTEGER FK | Посилання на батарею |
| voltage | DECIMAL | Напруга (В) |
| current | DECIMAL | Струм (А) |
| charge_level | DECIMAL | Рівень заряду (%) |
| temperature | DECIMAL | Температура (°C) |
| timestamp | TIMESTAMP | Час вимірювання |
| measured_by | INTEGER FK | Оператор |

**Таблиця Incidents (Інциденти)**
| Поле | Тип | Опис |
|------|-----|------|
| id | INTEGER PK | Унікальний ідентифікатор |
| battery_id | INTEGER FK | Посилання на батарею |
| measurement_id | INTEGER FK | Посилання на вимірювання |
| incident_type | VARCHAR(50) | Тип інциденту |
| description | TEXT | Опис інциденту |
| severity | VARCHAR(20) | Серйозність (low, medium, high, critical) |
| is_resolved | BOOLEAN | Вирішено |
| created_at | TIMESTAMP | Дата створення |
| resolved_at | TIMESTAMP | Дата вирішення |

### 3.3. Нормалізація

База даних нормалізована до третьої нормальної форми (3NF):

**1NF (Перша нормальна форма):**
- Всі атрибути є атомарними
- Немає повторюваних груп

**2NF (Друга нормальна форма):**
- Відсутні часткові функціональні залежності
- Всі неключові атрибути залежать від всього первинного ключа

**3NF (Третя нормальна форма):**
- Відсутні транзитивні залежності
- Неключові атрибути залежать тільки від первинного ключа

### 3.4. SQL запити

Система містить 10 складних SQL запитів для аналітики:

1. **Середній рівень заряду батарей по обладнанню**
2. **Пошук батарей з критичною температурою**
3. **Історія вимірювань батареї з агрегацією по годинам**
4. **Статистика інцидентів по типах і місяцях**
5. **Середня напруга батарей з відхиленням від номіналу**
6. **Обладнання з найбільшою кількістю інцидентів**
7. **Максимальна температура батареї з деталізацією**
8. **Мінімальний рівень заряду з прогнозом розряду**
9. **Середній струм з групуванням по діапазонах**
10. **Кількість вимірювань по обладнанню з активністю операторів**

## 4. Реалізація системи

### 4.1. Архітектура системи

Система побудована за клієнт-серверною архітектурою з трьома рівнями:

```
┌─────────────────┐
│  React Frontend │  ← SPA додаток (TypeScript, Tailwind CSS)
│  (Порт 3000)    │
└────────┬────────┘
         │ HTTP/JSON
         ▼
┌─────────────────┐
│  FastAPI Backend│  ← REST API (Python, SQLAlchemy)
│  (Порт 8000)    │
└────────┬────────┘
         │ SQL
         ▼
┌─────────────────┐
│   PostgreSQL    │  ← Реляційна БД
│   (Порт 5432)   │
└─────────────────┘
```

### 4.2. Backend реалізація

**Структура проекту:**
```
backend/
├── alembic/              # Міграції бази даних
├── models/
│   ├── models.py         # SQLAlchemy моделі
│   └── schemas.py        # Pydantic схеми
├── routers/
│   ├── equipment.py      # API обладнання
│   ├── batteries.py      # API батарей
│   ├── measurements.py   # API вимірювань
│   ├── incidents.py      # API інцидентів
│   └── reports.py        # API звітів
├── tests/                # Тести (pytest)
├── database.py           # Конфігурація БД
├── main.py              # Точка входу
└── requirements.txt      # Залежності
```

**Основні компоненти:**

1. **Моделі (SQLAlchemy):**
   - Визначення таблиць та відношень
   - Типи даних та обмеження
   - Індекси для оптимізації

2. **Схеми (Pydantic):**
   - Валідація вхідних даних
   - Серіалізація відповідей
   - Документація API

3. **Роутери (FastAPI):**
   - CRUD операції
   - Фільтрація та сортування
   - Обробка помилок

4. **Звіти:**
   - 9 аналітичних ендпоінтів
   - Агрегація даних
   - Візуалізаційні дані

### 4.3. Frontend реалізація

**Структура проекту:**
```
frontend/
├── src/
│   ├── components/       # React компоненти
│   │   ├── Layout.tsx
│   │   └── Modal.tsx
│   ├── pages/           # Сторінки
│   │   ├── Dashboard.tsx
│   │   ├── EquipmentPage.tsx
│   │   ├── BatteriesPage.tsx
│   │   ├── MeasurementsPage.tsx
│   │   ├── IncidentsPage.tsx
│   │   └── ReportsPage.tsx
│   ├── services/        # API сервіси
│   │   └── api.ts
│   └── types/           # TypeScript типи
├── tests/              # Тести (Vitest)
└── package.json
```

**Основні компоненти:**

1. **Layout:**
   - Навігаційне меню
   - Адаптивний дизайн (мобільний/десктоп)
   - Іконки Lucide React

2. **Сторінки:**
   - Dashboard — оглядова панель
   - EquipmentPage — управління обладнанням
   - BatteriesPage — управління батареями
   - MeasurementsPage — вимірювання
   - IncidentsPage — журнал інцидентів
   - ReportsPage — звіти з графіками

3. **Форми:**
   - Валідація на стороні клієнта
   - Обробка помилок сервера
   - Модальні вікна

4. **Графіки (Recharts):**
   - Лінійні графіки розряду
   - Температурні графіки
   - Статистичні діаграми

### 4.4. Валідація даних

**Backend валідація (Pydantic):**
- Температура: максимум 80°C
- Рівень заряду: від 0 до 100%
- Напруга: більше 0
- Струм: не може бути від'ємним
- Обов'язкові поля: name, type, serial_number

**Frontend валідація:**
- HTML5 валідація форм
- Перевірка перед відправкою
- Відображення помилок

### 4.5. Автоматизація інцидентів

Система автоматично створює інциденти при:
- Температурі > 60°C (критична)
- Температурі > 70°C (аварійна)
- Рівні заряду < 20% (низький)
- Рівні заряду < 10% (критичний)

## 5. Інтерфейс користувача

### 5.1. Навігація

Інтерфейс містить бічне меню з розділами:
- Dashboard — оглядова панель
- Оборудование — управління обладнанням
- Батареи — управління батареями
- Измерения — введення вимірювань
- Инциденты — журнал подій
- Отчеты — аналітичні звіти

### 5.2. Форми введення

**Форма обладнання:**
- Назва (обов'язкове)
- Тип (обов'язкове)
- Розташування (обов'язкове)
- Опис (опціональне)
- Статус (випадаючий список)

**Форма батареї:**
- Обладнання (вибір зі списку)
- Серійний номер (обов'язкове, унікальне)
- Ємність (Ah)
- Номінальна напруга (V)
- Дата встановлення
- Статус

**Форма вимірювання:**
- Батарея (вибір зі списку)
- Напруга (V)
- Струм (A)
- Рівень заряду (%)
- Температура (°C)

### 5.3. Звіти та графіки

**Доступні звіти:**
1. **Рівень заряду** — середні показники по обладнанню
2. **Температурні сповіщення** — батареї з високою температурою
3. **Відхилення напруги** — аналіз відхилень від номіналу
4. **Статистика інцидентів** — по типах і місяцях
5. **Інциденти по обладнанню** — топ обладнання з проблемами
6. **Статистика струму** — розподіл за діапазонами
7. **Активність вимірювань** — по операторах
8. **Графік розряду** — історія заряду батареї
9. **Температурний графік** — зміни температури
